## 발견된 버그

### Bug 1: short-interest Yahoo 크럼 쿠키 형식 오류 (CRITICAL)

**문제:**
`getYahooCrumb()` 함수가 `headers.get('set-cookie')` 사용.
이 메서드는 복수 Set-Cookie 헤더를 콤마로 이어붙여 반환 —
결과값에 `Path=/; Domain=.yahoo.com; Expires=...` 등이 포함됨.
Cookie 요청 헤더로 이 값을 그대로 보내면 Yahoo가 인증 거부 → 크럼 획득 실패 → Yahoo v10 데이터 null.

**추가 문제:** CRUMB_TTL = 1h (sector-pe의 22h와 불일치)
→ sector-pe가 22h 크럼을 Redis에 저장해도, short-interest가 1h TTL로 덮어씀
→ 크럼이 1h마다 갱신 시도 → 불필요한 Yahoo 홈페이지 요청 + 오염 위험

**근본 원인:**
`headers.get('set-cookie')` vs `headers.getSetCookie()` 메서드 차이.
`getSetCookie()`는 각 Set-Cookie를 독립 배열로 반환 → name=value 파트만 추출 가능.

**수정 (commit 0478987):**
```typescript
// BEFORE (buggy):
const CRUMB_TTL = 60 * 60; // 1h
const cookie = homeRes.headers.get('set-cookie') ?? '';
if (!crumb) return null;
await loggedRedisSet(redis, 'api.short-interest', CRUMB_KEY, result, { ex: CRUMB_TTL });

// AFTER (fixed):
const CRUMB_TTL = 22 * 60 * 60; // match sector-pe — crumbs last ~24h
const rawCookies = homeRes.headers.getSetCookie?.() ?? [];
const cookie = rawCookies
  .map(c => c.split(';')[0])
  .filter(c => c.startsWith('A1=') || c.startsWith('A3=') || c.startsWith('A1S='))
  .join('; ');
if (!cookie) return null;
if (!crumb || crumb.startsWith('{')) return null;  // JSON 오염 방어
await loggedRedisSet(redis, 'api.short-interest', CRUMB_KEY, result, { ex: CRUMB_TTL });
```

**사용자 영향 (수정 전):**
- shortFloatPct, shortRatio, trailingPE: Yahoo v10 데이터 null 가능성
- sector-pe의 22h 크럼 캐시를 1h TTL로 덮어쓸 위험

**사용자 영향 (수정 후):**
- Yahoo v10 데이터 정상 수신 (sector-pe와 동일 로직)
- Redis 크럼 TTL 22h로 통일 → 공유 키 충돌 없음
