2026-05-15 14B 모델 업그레이드 — TabbyAPI 실패, Ollama Q3 채택
================================================================

목적: qwen3:8b → qwen2.5:14b 업그레이드로 보고서 품질 향상.
배경: 누적 158 outcome 분석 결과 'high' confidence hit rate 15%→11% 악화.
모델 자체가 의사결정 병목 (Codex 분석 후 결론).

시도 1: TabbyAPI + ExLlamaV2 (EXL2 3.5bpw) — 실패
---------------------------------------------------
설치: scripts/setup-tabbyapi.ps1
- PowerShell 5.1 `2>&1 | Out-Null` 가 pip stderr 를 NativeCommandError 로 wrap
  → ErrorActionPreference=Stop 와 결합해 즉시 abort. 수동 pip 직접 호출로 해결.
- .tabbyapi 가 pyproject.toml 사용 (requirements.txt 아님) → `pip install -e .[cu12]`
  로 cu128 torch + exllamav2 + flash_attn 일괄 설치.

모델: turboderp/Qwen2.5-14B-Instruct-exl2 (원본 repo 부재) → bartowski 3_5 branch
사용. output.safetensors 7.95 GB 단일 파일 (HF index 의 8 shard 는 EXL2 무관).

서버 기동 결과: torch.OutOfMemoryError at 82% (81/99 modules loaded)
- RTX 4050 6GB VRAM 에 14B EXL2 3.5bpw 안 들어감.
- CLAUDE.md "EXL2 3.5bpw = ~6 GB, GPU only" 추정은 낙관적 — 실측 7.95GB.
- TabbyAPI/EXL2 는 CPU offload 미지원 (GPU only 정책).

결론: 14B EXL2 on 6GB VRAM 인프라 자체가 인feasible.

시도 2: Ollama Q3_K_M (GGUF, llama.cpp 백엔드) — 성공
-----------------------------------------------------
`ollama pull qwen2.5:14b-instruct-q3_K_M` (7.3 GB)
- llama.cpp 가 자동 CPU+GPU offload 처리 → 6GB VRAM 에서 작동
- 속도: 8B 대비 2-3x 느림 (Wave1 5병렬 ~10-15분 예상)
- 품질: 14B 의 의사결정 능력 + EXL2 보다 5-10% 낮은 정확도 (Q3 양자화 손실)
