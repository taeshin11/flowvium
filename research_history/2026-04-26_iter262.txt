- lock loser 대기: 4s → 캐시 없음 → AI 분석 직접 실행
- 결과: 피크 시간 (자정 UTC = KST 9AM) 다수 Lambda가 동시에 AI 호출

### 수정 내용
```typescript
// Before:
if (!acquired) {
  await new Promise(r => setTimeout(r, 4000));
  const fresh = await redis.get<NewsWithCascade[]>(listKey());
  if (fresh && fresh.length > 0) return ...;
  // Lock holder may have crashed — proceed anyway
}

// After:
if (!acquired) {
  // Poll up to 30s; returns as soon as list is ready
  for (let i = 0; i < 6; i++) {
    await new Promise(r => setTimeout(r, 5000));
    const fresh = await redis.get<NewsWithCascade[]>(listKey());
    if (fresh && fresh.length > 0) return ...;
    const stillLocked = await redis.get(LOCK_KEY);
    if (!stillLocked) break; // holder finished or crashed
  }
  // Final check + fall-through if crashed (same safety net)
}
```

### 왜 이 설계인가
- 락 홀더는 list 먼저 쓰고(L496) lock 해제(L509) → lock 없음 = list 있음
- 6 × 5s = 30s 폴링은 maxDuration=60s 내에 충분히 여유
- Hobby plan (maxDuration=10s) 영향 없음: Redis 없으면 lock 코드 진입 안 함
