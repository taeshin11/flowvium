## 남은 false-positive — earnings.ALL "endpoint stale"
### 재구성된 원인
`/api/earnings` 라우트는 `if (!FINNHUB_KEY) → return { warning: "FINNHUB_KEY 미설정..." }`.
- 외부 curl: Redis cached 응답 반환 (warning 없음, 정상 데이터)
- verify-metrics 내부 fetch(alias/self): 일부 edge instance가 env 없는 배포로 라우팅 → warning 필드 포함 응답
  → 15분 이상 지속 — 단순 propagation 레이스 아님, 구조적 Vercel alias routing 이슈

즉, 내부 fetch 기반 헬스체크는 근본적으로 신뢰할 수 없음 (멀티 배포 환경).

## 수정 (src/app/api/cron/verify-metrics/route.ts)
`verifyEarnings()` 을 Finnhub 직통 프로브로 전환:
- `fetch('https://finnhub.io/api/v1/calendar/earnings?from=today&to=+14d&token=KEY')`
- 8s timeout, 500ms 이하 빠름
- 응답의 earningsCalendar 배열 길이로 ok/degraded 판단
- 429 → degraded (rate limit), 그 외 non-2xx → error

이점:
1. Vercel alias routing 의존 제거 — 내부 fetch 사용 안 함
2. /api/earnings 자체는 Redis 2h 캐시 → Finnhub 직통 호출은 크론마다 1회 (30분 주기) → 60 req/min 티어 여유
3. /api/earnings 라우트가 다운되어도 earnings.ALL 헬스는 정확하게 보고

## 남은 false-positive 가능성
동일 패턴(env-dependent silent fallback)인 다른 엔드포인트 조사 — 확인 결과 없음:
- capital-flows, macro, fedwatch, fg, credit 는 Redis 공유 데이터 기반 → 배포 간 동일 응답
- insider, market: 환경 변수 의존 없음

즉 이 패턴은 earnings 단일 케이스.
